Optional Dictionary Keys

Status: implemented on optional_dict_keys_typing_design. Optional shape fields (name?: T), the always-fails unwrap diagnostic, and the re-typed option-dict builtins (numFmt, httpGet/httpPost, grid groupBy aggs, zip*) all ship; access semantics are unchanged as designed. Full test suite (typecheck, runtime, go test) passes.

Optional keys are a property of a dictionary shape type only. They change how a value binds against a declared shape (function signatures, as casts). They do not change getters, ?, none, or Maybe — those already represent the runtime faithfully.

Motivation

Shapes today require every listed field to be present, so APIs with optional keys can't be typed precisely. The only fallback is the loose homogeneous dict {v} ({str: v}), which discards all per-field information. This is the documented workaround in the source for httpGet ("the type system has no notion of optional shape fields"). Real cases: numFmt options (all optional), httpGet/httpPost requests (url required, rest optional), grid groupBy agg specs (agg required, name/meta optional), the zip* options dicts, and JSON/spreadsheet records.

Syntax

Mark an optional field with ? after the field name:

type Req = {url: str, timeout?: int, headers?: {str: str}}

In def signatures (historical quoted-field spelling), the ? follows the quote:

def doGet ({ "url": str, "timeout"?: int } -- str) :url? end

The marker sits between name and colon: name?: T. FormatType renders it the same way, so errors read back as written.

? marks the key as possibly-absent. This is orthogonal to Maybe, which marks the value. Both compose:

TypeMeaning
timeout?: intkey may be absent; if present, value is int
timeout: Maybe[int]key always present; value is just int or none
timeout?: Maybe[int]key may be absent; if present, just int or none

What optional keys do: subtyping only

The feature lives entirely in shape unification — i.e. when a value is bound or passed against a declared shape. For each field f of the expected shape:

expected factual has it (required)actual has it (optional)actual lacks it
f: T (required)unify value typesreject (may be absent)reject
f?: T (optional)unify value typesunify value typesaccept

Extra fields in the actual value are always allowed (existing width subtyping). The two asymmetries: a required value satisfies an optional parameter; an optional value does not satisfy a required parameter. Critically, a present optional field is still type-checked — that is the win over {v}: { "timeout": "soon" } is rejected against {timeout?: int} because str ≠ int.

Prototype results (full suite passed with zero edits to tests or stdlib):

  • omit an optional key when building/passing → accepted
  • present optional key, wrong type → rejected
  • present optional key, right type → accepted
  • missing a required key → rejected
  • required value → optional parameter → accepted; optional value → required parameter → rejected

What optional keys do NOT do: access is unchanged

Getters already represent the runtime, so nothing here changes:

  • :k (the getter) never fails at runtime — it returns a Maybe. So its type is Maybe[T], for required and optional fields alike.
  • ? is fromJust: it unwraps Just, fails on None. So :k? is T, a partial operation. The checker types the success case; it does not promise the unwrap won't fail — exactly as fromJust Nothing :: a type-checks in Haskell.

So required vs optional fields differ only in subtyping/construction, never in how you read them. There are no access-site type errors, and none/Maybe/? keep their faithful ADT meaning.

Relationship to Haskell's Maybe

data Maybe a = Just a | Nothing introduces one type (Maybe) and two values (Just, Nothing). Nothing :: Maybe a — there is no type called Nothing. mshell matches this for values: none : ( -- Maybe[a]) is Nothing, just : (a -- Maybe[a]) is Just. A getter returning Maybe[T] and ? being a partial fromJust is the faithful model. (mshell historically overloaded none as both this constructor and a JSON-null type; that split has since been done — none is now only the constructor, and null is the JSON null type. See "Related" below. This also confirms why an earlier "type the absent getter as none" idea was rejected: it only worked by being un-ADT-like.)

LSP diagnostic: "this unwrap will always fail"

Faithful types don't say whether a given Maybe is Just or Nothing, so the "always fails" signal is a non-fatal diagnostic (SeverityInfo, the severity dbg already uses — it never fails --check-types), not a type error.

The "always None" fact is encoded in the value's type, so it rides through variable bindings and arbitrary stack flow for free, and the failure is reported at the ? — exactly where it happens at runtime. A getter for a field a concrete shape does not declare yields Maybe[bottom]: still a real Maybe (so ?, match, and maybe all type-check, and bottom unifies with anything downstream — access stays faithful), but its Just payload is uninhabited, meaning the value can only be None. The ? handler inspects its operand: if it is Maybe[bottom], the unwrap is statically guaranteed to fail and a SeverityInfo hint is emitted on the ? token. This is why { 'a': 1 } :b val! @val ? is flagged even though the getter and the unwrap are far apart — the Maybe[bottom] type flows through val.

The element type is fully determined by the receiver kind and field status — the concrete-vs-open distinction is exactly shape (TKShape) vs homogeneous dict (TKDict), which the type system already tracks:

receiverfield:k getter type? diagnostic
homogeneous dict {str: T}any keyMaybe[T]never
shape, required fielddeclaredMaybe[T]never (safe)
shape, optional fielddeclared k?:Maybe[T]never ("always" can't be claimed)
shape, undeclared fieldnot in shapeMaybe[bottom]yes

Rule: ? applied to a Maybe[bottom] operand emits the diagnostic. Open dictionaries return the genuine Maybe[T] for any key, so they are never hinted — a statement is only possible against a concrete shape.

The hint is contract-flavored, not an absolute claim about every runtime value: width subtyping means a wider value could carry an undeclared field, but a value typed {a: int} only guarantees a, so :b? cannot be relied on (a function that needs b should declare it). The hint is placed on the ? token: '?' unwraps a value that can only be None (e.g. a getter for a field the shape does not declare); this will fail at runtime.

Implementation

The feature itself is one small, self-contained change set:

  1. Type.go: add Optional bool to ShapeField; include it in encodeShapeKey so {k: T} and {k?: T} intern distinctly. (Sorting/dedup unaffected.)
  2. TypeExpr.go: add Optional bool to TypeShapeField; accept ? between a field name and its colon (fast path and continueTypeShapeBody, covering quoted names); propagate in resolveTypeExpr. (No lexer change — QUESTION already tokenizes; no value-parser/getter change.)
  3. TypeError.go: render ? on optional fields in FormatType.
  4. TypeChecker.go: extend unifyShape with the subtyping table above. Casts inherit it via castOk → unify.

The diagnostic is a separable add-on with no lookahead. lookupGetterValueType returns bottom for an undeclared concrete-shape field, so the getter pushes Maybe[bottom]; the ? handler in checkOne inspects its operand and emits a SeverityInfo when it is Maybe[bottom]. Because the signal lives in the operand's type, it survives variable bindings and arbitrary stack flow and is reported at the ? itself. Duplicate hints from branch fan-out (or a re-walked body) are collapsed by position. Surfaced by lsp.go; ignored by pass/fail.

Supporting work (per CLAUDE.md): re-type the motivating builtins (numFmt, httpGet/httpPost, grid groupBy aggs, zip* options) from {v} to honest optional shapes; docs (doc/type_system.inc.html, doc/mshell.md, rebuild); CHANGELOG.md under Unreleased/Added.

Approaches considered and rejected

  • Merge :k? into one "checked getter" at parse time. Worked and passed the suite, but changes the value parser/AST and absorbs ? (a uniform standalone operator) into the getter. Unnecessary once access stays faithful.
  • Type an absent getter as the none primitive so ? rejects it. Catches the case with no parser change, but only by making the result un-Maybe — it breaks the ADT (a missing field is Nothing : Maybe a) and the safe match/maybe accessors. Rejected.
  • Make undeclared-field access a hard type error. Conflicts with the guiding principle: the runtime returns None (it does not fail), so the type system must not fail either. Demoted to the non-fatal diagnostic above.

Related (done): the distinct null type

Status: implemented and merged (PR #258, "Add null"). This section records the outcome; it is no longer a proposal.

Previously one word "none" did two jobs over one runtime object (Maybe{nil}): the Maybe Nothing value, and a standalone none type used for JSON null in unions like int | str | none. Those are different concepts — {"x": null} has x present with a null value; {} has x absent — and the overlap was the root of much confusion (it is why the old none type was not a Maybe).

That split is now in place:

  • null is a first-class primitive type and value for the JSON null. The null literal pushes one (( -- null)), parseJson produces null for JSON null, it can appear in unions (int | null), and it can be matched with a null arm.
  • none is now only the empty-case constructor of Maybe. It is no longer a nameable type — using none in a type expression is an error. So Maybe/none is a clean optionality ADT.
  • int | null (an integer or a literal JSON null) is deliberately distinct in meaning from Maybe[int] (an integer that may be absent). The canonical JSON scalar is now type JsonScalar = str | int | float | bool | null.

This directly benefits the design above. The Maybe/none story is now unambiguous, and the "always-fails" diagnostic's NoneKnown tag means absence only — a value of type null is a real value and is never confused with Nothing, so the diagnostic can never misfire on legitimately-readable JSON nulls.
