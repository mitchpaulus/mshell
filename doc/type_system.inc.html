The mshell type system checks stack effects before a script runs. It does not change the runtime data model described in Data Types; instead it gives the checker enough information to prove that values, variables, quotations, branches, and definitions agree about what is on the stack.

Running the Checker § Back to top

Use --check-types to type-check a file. When type checking succeeds, the script continues to normal execution. When type checking fails, execution is stopped before user code runs. Use --type-check-only to run the same checker and exit without evaluating the script. 

msh --check-types script.msh
msh --type-check-only script.msh

The checker reads definitions from the standard library and the current file. Definition signatures are the main source of static information. Literal expressions, variable stores and retrieves, built-ins, quotations, and control-flow constructs are then checked against those signatures.

Stack Effects § Back to top

Function signatures use a stack-effect form: 

def name (inputs -- outputs)
    ...
end

Inputs appear in the order they must already be on the stack. The rightmost input is the top item consumed first by the definition body. Outputs appear in the order left on the stack after the definition finishes. 

def addOne (int -- int)
    1 +
end

def fullName (str str -- str)
    last!, first!
    $"{@first} {@last}"
end

Generic type names are written as ordinary lowercase or uppercase names in signatures. The same name means the same type within one signature. This is how list and higher-order functions preserve relationships without naming a concrete type. 

def first ([a] -- a)
    :0:
end

def applyTwice (a (a -- a) -- a)
    f!
    @f x
    @f x
end

Type Expressions § Back to top

The primitive static type names are int, float, bool, str, path, datetime, bytes, and none. Named runtime types such as Grid, GridView, and GridRow are also available.

Form Meaning Example
T Primitive, named type, alias, or generic int, str, Person, a
[T] List whose elements all fit type T [str], [int | str]
{K: V} Dictionary from key type K to value type V {str: int}
{name: T, age: U} Dictionary shape with known fields {name: str, age: int}
Maybe[T] Optional value, usually returned by lookups and parsing Maybe[int]
T | U Union: a value may be any listed type int | float
(A B -- C) Quotation type with its own stack effect (int int -- bool)

Top-level type declarations give names to type expressions. They are useful for casts and for naming larger shapes and unions that are reused by other type declarations. Current definition signatures still use the historical signature parser, so dictionary shapes in def signatures are written with quoted field names as shown below. 

type Person = {name: str, age: int}
type JsonScalar = str | int | float | bool | none
type JsonDict = {str: JsonScalar}

{ "name": "Ada", "age": 36 } as Person :age? 1 +

Use as to make an explicit assertion when the checker needs help with a value that cannot be inferred precisely enough from local syntax. 

{ "name": "Ada", "age": 36 } as Person

Dictionaries § Back to top

Runtime dictionaries have string keys. The type system can describe dictionaries in two complementary ways: as homogeneous maps, where every key has the same value type, or as shapes, where specific keys have specific value types.

Homogeneous Dictionaries §

A homogeneous dictionary is best when the keys are dynamic and every value has the same meaning. For example, a count table is a string-keyed dictionary of integers. 

{ "passed": 10, "failed": 2 } as {str: int} values len

Older definition signatures also support a compact dictionary form: 

def countValues ({ int } -- int)
    values len
end

In a definition signature, { int } means a dictionary whose string keys all have int values. The equivalent type-expression spelling is {str: int}. The wildcard spelling { *: int } is also accepted in definition signatures and has the same practical meaning.

Dictionary Shapes §

A shape is best when a dictionary is being used as a record. Each named field has its own static type, so the checker can validate field access and prevent accidental field/value mixups. 

def labelPerson ({ "name": str, "age": int, "active": bool } -- str)
    person!
    @person :name? name!
    @person :age? age!
    $"{@name} ({@age})"
end

The :field? style is the usual way to read a required field from a checked shape. Without ?, dictionary lookup returns Maybe[T]. The question mark unwraps the Maybe and preserves the plain field type for the rest of the expression. 

type Person = {name: str, age: int}

{ "name": "Ada", "age": 36 } as Person :age? 1 +

Nested Dictionaries §

Dictionary shapes compose with lists, unions, and homogeneous dictionaries. This makes JSON-like data much more precise than a single broad dict type. 

type Address = {city: str, state: str}
type Person = {
    name: str,
    age: int,
    address: Address,
    tags: [str]
}

{
    "name": "Ada",
    "age": 36,
    "address": {"city": "London", "state": "UK"},
    "tags": ["math" "computing"]
} as Person :address? :city?

Use homogeneous dictionaries inside shapes when one field is itself a dynamic map. 

type Report = {
    title: str,
    totals: {str: float},
    notes: [str]
}

Lists § Back to top

Runtime lists can hold any mix of values. The static type system describes that in one of two useful ways: as a homogeneous list with a single element type, or as a list whose element type is a union.

Homogeneous Lists §

A homogeneous list has one element type. This is the strongest and most useful list type because functions such as map, filter, sortByCmp, zip, foldl, and indexing can preserve the element type exactly. 

def doubleAll ([int] -- [int])
    (2 *) map
end

def names ([{ "name": str, "age": int }] -- [str])
    (:name?) map
end

The signature [a] means "a list of some single element type". It does not mean "any list with unrelated element types". This lets generic functions keep their input and output relationships precise. 

def keepPassing ([a] (a -- bool) -- [a])
    filter
end

Heterogeneous Lists §

When a list intentionally contains a fixed set of possible element kinds, use a union element type. For example, [int | str] is a list where every element is either an integer or a string. 

type Cell = int | float | str | bool
type Row = [Cell]
type Table = [Row]

[1 "Ada" true] as [int | str | bool]

This is powerful for shell data because many real inputs are table-like or JSON-like rather than class-like. A CSV row, a spreadsheet row, a parsed JSON array, or command output that has been split into fields can be represented without discarding the known set of legal element types. 

type CsvCell = int | str
type CsvRow = [CsvCell]
type CsvRows = [CsvRow]

[["Ada" 36] ["Grace" 42]] as [[int | str]] (len 0 >) filter

The checker currently treats heterogeneous lists as lists of unions, not as fixed-length tuples with per-index types. That means [1 "Ada"] can be understood as [int | str], but the checker does not use index :0: to prove "this position is always int" unless the value is converted or asserted in another way.

Quotations § Back to top

Quotation types describe the stack effect of code values. This is what makes higher-order definitions such as map, filter, and sortByCmp checkable. 

def adults ([{ "name": str, "age": int }] -- [{ "name": str, "age": int }])
    (:age? 18 >=) filter
end

The quotation (:age? 18 >=) is inferred as a function from a person-shaped dictionary to bool because filter expects a predicate quotation. Context is important: many quotations start generic and become concrete only when passed to a function or checked against a definition signature.

Operators with multiple valid signatures keep an overload set until context resolves them. For example, the quotation (>) is not immediately "integer greater-than". It can become (int int -- bool), (float float -- bool), (str str -- bool), or (datetime datetime -- bool) depending on the expected quotation type. 

def after (datetime datetime (datetime datetime -- bool) -- bool)
    cmp!
    @cmp x
end

2024-01-02 2024-01-01 (>) after

Control Flow § Back to top

Branching constructs must reconcile stack and variable state across reachable branches. For if, else, iff, and match, each non-diverging branch must leave the same number of stack items. Corresponding stack positions are unified so compatible branch outputs become a shared type or union. 

true if
    1
else
    2
end
# leaves int

true if
    1
else
    "one"
end
# leaves int | str

Variables bound inside branches also have to agree. If one reachable branch binds name!, the other reachable branches must bind name! too, and the bound types must be compatible.

Branches that diverge with return, break, or continue are excluded from reconciliation. That lets guard-style code type-check naturally. 

def positiveName (int -- str)
    n!
    @n 0 < if
        "negative" return
    end
    "positive"
end

Current Boundaries § Back to top

The checker is intentionally static and conservative. It catches stack underflows, unknown definitions, incompatible built-in usage, incompatible branch results, and definition bodies that do not match their declared signatures. Runtime checks still exist for data-dependent conditions such as missing dictionary keys, failed ? unwraps, file-system failures, process exit codes, and values that come from external programs.

When the checker cannot infer a precise shape for dynamic data, prefer adding a named type and an as assertion near the boundary where the data enters typed code. That keeps the rest of the script checked with useful dictionary, list, union, and quotation types.
